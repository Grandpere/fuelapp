# UI i18n Migration Design

## Goal
Move the application UI to Symfony translations with French as the default locale and English as the fallback locale, starting with the main user-facing interface and deferring admin/back-office to a second wave.

## Product direction
- The product targets France first, so the visible application language should be French by default.
- English should remain available as a technical fallback for untranslated keys during the migration.
- There is no need for a visible language switch in v1.
- The first problem to solve is mixed hard-coded copy, not runtime language choice.

## Scope

### In scope for wave 1
- Symfony translator configuration.
- Default locale set to `fr`.
- Enabled locales limited to `fr` and `en`.
- English fallback configured for untranslated keys.
- New translation catalog files for shared UI copy.
- Migration of the main user-facing web UI:
  - shared layout and navigation,
  - common actions and recurring labels,
  - dashboard,
  - receipts,
  - imports,
  - vehicles,
  - stations,
  - maintenance,
  - analytics.
- Functional test updates required by translated UI copy.

### Out of scope for wave 1
- Admin/back-office screens.
- API payload localization.
- Domain exception localization unless already surfaced directly in the user UI.
- External data source localization.
- A user-visible locale switcher.

## Non-goals
- Do not translate code identifiers, route names, service names, entity fields, or internal logs.
- Do not rewrite all product copy at once beyond what is needed for the migrated screens.
- Do not translate externally sourced data such as public fuel station labels coming from imported datasets.

## Current state
- The project currently has no dedicated translation configuration file and no `translations/` directory.
- Twig templates mostly render hard-coded English strings, with a few French or mixed-language strings.
- Functional tests often assert exact rendered labels, so translation work will affect test expectations.

## Recommended approach
Adopt Symfony translations now, but migrate screens incrementally by area instead of attempting a full repository-wide conversion in one pass.

### Why this approach
- It fixes the user-visible language problem quickly on the primary product surfaces.
- It keeps the diff size reviewable.
- It reduces the chance of breaking a large number of tests at once.
- It establishes stable translation conventions before repeating the work in admin/back-office.

## Locale behavior

### Default behavior
- Application default locale: `fr`
- Enabled locales: `fr`, `en`
- Translator fallback: `en`

### Runtime behavior
- If a translated French key exists, the user sees French.
- If a French key is missing but an English key exists, Symfony falls back to English.
- If both are missing, the raw translation key appears, which should be treated as a bug and covered by review/testing.

### Language switch
- No explicit language selector in the UI for now.
- If locale negotiation is later needed, it can be added on top of the same translation catalogs without redoing the copy migration.

## Translation file strategy

### Initial files
- `translations/messages.fr.yaml`
- `translations/messages.en.yaml`

### Key naming
Use semantic, reusable keys rather than template-specific or sentence-as-key patterns.

Examples:
- `nav.dashboard`
- `nav.receipts`
- `action.save`
- `action.reset`
- `action.cancel`
- `maintenance.title`
- `maintenance.empty.events`
- `receipt.list.title`

### Key organization
Group keys by user-facing concern:
- `nav.*` for global navigation
- `action.*` for reusable actions
- `common.*` for shared generic labels
- `dashboard.*`, `receipt.*`, `import.*`, `vehicle.*`, `station.*`, `maintenance.*`, `analytics.*` for screen-specific text

This keeps shared wording centralized while allowing feature-specific text to remain readable.

## Migration strategy

### Step 1 - Infrastructure
- Enable and configure Symfony translation support.
- Set `default_locale: fr`.
- Add enabled locales and fallback locales.
- Create the initial translation files.

### Step 2 - Shared shell
- Migrate topbar navigation, page-level recurring actions, empty-state primitives, and common flash labels first.
- This gives the app a coherent French baseline across multiple screens immediately.

### Step 3 - User-facing screens
Migrate templates by product area in this order:
1. dashboard
2. receipts
3. imports
4. vehicles
5. stations
6. maintenance
7. analytics

This order prioritizes the most visible and frequently used surfaces.

### Step 4 - Admin wave
- Reuse the same translation structure for admin/back-office screens in a separate follow-up task.
- Keep admin strings hard-coded until that wave starts, relying on the same translator setup already in place.

## Template usage rules
- All new user-visible strings in migrated areas must go through translation keys.
- Prefer Twig `|trans` or `trans()` consistently with existing project conventions once introduced.
- Reuse shared keys for common actions instead of duplicating near-identical copy across feature namespaces.
- Keep interpolation parameters explicit for counts, names, or contextual values.

## Testing strategy

### Functional tests
- Update functional tests that assert exact UI strings on migrated screens.
- Where the user-visible copy is the behavior under test, assert the expected French text.
- Where exact wording is incidental, prefer more stable assertions on structure, links, forms, or semantic blocks.

### Regression focus
- Ensure migrated screens still render successfully.
- Ensure no raw translation keys are rendered in place of missing copy.
- Ensure common navigation and action labels stay consistent across pages.

### Quality gate expectations
- Run the normal project quality gates after implementation.
- If broad UI copy changes affect functional tests, the user should still run the manual functional suite as defined by project rules.

## Risks and mitigations

### Risk: mixed translated and hard-coded copy during migration
- Mitigation: migrate the shared shell first, then complete one user area at a time.

### Risk: fragile tests due to exact text assertions
- Mitigation: keep wording assertions where meaningful and reduce incidental string matching elsewhere.

### Risk: key sprawl or inconsistent naming
- Mitigation: establish shared namespaces (`nav`, `action`, `common`) before feature-specific catalogs grow.

### Risk: untranslated strings slipping through
- Mitigation: review diffs for remaining hard-coded UI strings in migrated templates and watch for raw keys in rendered output.

## Open decisions already resolved
- Product default language: French
- Fallback language: English
- Visible language switch in v1: no
- Migration shape: progressive by area
- Admin timing: second wave after user-facing UI

## Implementation readiness
This design is ready to be turned into an implementation plan focused on:
- translation infrastructure,
- shared UI migration,
- user-facing screen migration,
- test adaptation,
- follow-up admin migration preparation.
